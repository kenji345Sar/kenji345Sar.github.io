業務システムの設計・実装・改修(直近)

期間: 約3ヶ月 技術: C# / ASP.NET Core / React / TypeScript 体制: 少人数チーム 担当領域: バックエンド + フロントエンド 両方

案件概要

業務システムの設計・実装・改修に参画。業務ルールや履歴管理を前提とした データ構造の整理から、API・画面側の改修、データ不整合の調査・修正まで、 バックエンドとフロントエンドをまたいで一連で担当した案件です。

課題

  • 要件が十分に整理されておらず、業務ルール・ドメイン概念から押さえる必要があった
  • 履歴や残高のように、時間軸とデータ整合性を両立させる必要がある業務領域
  • 一覧表示・取引履歴・計算結果などの整合性が崩れた状態のデータ調査・修正
  • 未経験の技術スタックを含む中で、短期間で形にすることが求められた

進め方

  • 業務ルールとドメインモデルの整理を起点に、実装に落とし込むための概念整理から着手
  • 業務側の要求があいまいな段階でも、画面・データ・処理の関係を確認しながら機能化
  • AIを活用しながら、未経験領域を含む設計検討・コード生成・改修案作成を並行
  • 不整合データの確認 → 原因切り分け → 修正方針の整理 → 反映 → 再確認 を一連で実施

担当範囲

フロント / バックエンドの 2領域(主要4技術スタック) にまたがって対応。複数の機能領域 について、ドメイン整理から実装・調査修正までを担当しました。

  • 業務領域に関するドメインモデル整理・設計検討(複数機能領域)
  • バックエンド(C# / ASP.NET Core)の改修、API修正、データ処理ロジック実装
  • フロントエンド(React / TypeScript)の画面修正、表示内容調整、操作フロー改善
  • 状態遷移とデータ整合性に関する調査・修正(不整合の原因切り分け → 修正方針整理 → 反映 → 再確認まで一連で対応)
  • フロントとバックエンド 2領域をまたいだ不具合調査・動作確認・修正対応

意識したこと

  • 要件が曖昧な状態でも止まらず、論点整理 → 試作 → 実装 → 修正まで小さく進める
  • 履歴管理や残高遷移など、後からズレると直しにくい領域は、最初にデータ構造で揃える
  • 既存業務・既存データを踏まえ、運用を意識した修正を選ぶ
  • AIで一気に生成するのではなく、業務ルールに合わせて取捨選択・調整する

今から見て

要件が十分に整理されていない状態から、ドメイン整理と実装を並行で進める 案件では、「何を作るか」より先に「どう崩れないようにするか」を決めることが 効果的でした。同じように、仕様が固まりきらない段階の業務システムに対して、 整理しながら短期で形にする関わり方ができます。
