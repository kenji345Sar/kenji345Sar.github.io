# 配送日報Web化 サンプル｜要件定義メモ

> このドキュメントは、Excel・紙運用のWeb化サンプルです。架空の中小企業（業務整理パックサンプルと同じ「株式会社マルキ食品」）を想定して作成しています。

## 想定の流れ（業務整理パックからの続き）

業務整理パックの結果、第1期「配送ルート知識のAI抽出」と並行して、**配送日報のWeb化**を先行で着手することになった、という想定です。

- 抽出対象であるベテラン配送員の日々の業務を、まず**構造化された形で蓄積**しておきたい
- 紙→Excel転記の手作業を**先に止めておく**ほうが、後段のAI活用が活きる
- 大規模パッケージは入れず、業務に合った小さい形で素早く立ち上げる

## 現状の運用（Before）

| 項目 | 内容 |
| --- | --- |
| 配送員の日報記入 | 手書きの紙（A4横、1日1枚／配送員1名） |
| 提出 | 帰社時に事務に手渡し |
| 集計 | 事務担当が翌朝Excelに転記（毎日30〜45分） |
| 保管 | 紙はバインダーで月別保管、Excelは月別ファイルで共有フォルダ |
| 活用 | 月次の集計でしか見ていない（個別日報は事実上、転記後は埋もれる） |

### 現状の主な不満

- 転記ミス（数量・配送先名）が月数件発生
- 「あの日の積み残し、どの便だっけ?」を探すのが大変
- 事務が休むと転記が溜まる
- ベテラン配送員の知見（道順・顧客の癖・荷下ろしの注意点）が紙に埋もれている

## このサンプルで作るもの（After）

- 配送員がスマホ／タブレットから直接入力できるWeb日報
- 入力された日報の**一覧・検索・CSV出力**ができる管理画面
- 月次集計向けのCSVは、現在Excelで使っている書式と互換にする
- AI活用ポイント（要約・検索・異常検知）は機能として用意するが、**まずは転記廃止と蓄積基盤の確立を優先**

## ゴール（数字目標）

| 指標 | 現状 | 目標 |
| --- | --- | --- |
| 事務の転記作業時間 | 30〜45分／日 | **0分**（廃止） |
| 転記ミスによる修正対応 | 月数件 | **月0件**（転記そのものを廃止） |
| 過去日報の検索時間 | 数十分〜数時間 | **30秒以内**（一覧・全文検索） |
| ベテラン日報の引き継ぎ用テキスト化 | できていない | **AI要約で日次・週次の自動生成** |

## 対応範囲

- 配送業務の日報入力・閲覧・検索・CSV出力
- 既存Excel書式と互換のCSV吐き出し
- ロール別の権限（配送員・事務・管理者）
- AI機能：日報要約、過去日報検索、異常値ハイライト

## 対応しない範囲（このフェーズでは）

- 配車・配送ルートの自動最適化（業務整理パックの第3期で扱う）
- 受注・請求システムとのリアルタイム連携（同・第3期）
- 営業日報、倉庫日報（**日報シリーズの拡張テンプレ**として後続で展開可能）

## 期間・進め方（標準）

- 期間：約6週間（要件確定〜本番運用開始）
- 工程：要件確定 → 画面・データ設計 → 実装 → 配送員2名で先行運用 → 全員展開
- **段階リリース**で第1版（入力＋一覧＋CSV）を最短2週間で出し、AI機能は第2版・第3版で追加

## 想定スタック（参考）

- フロント：HTML+最小限のJS（スマホブラウザで完結、アプリ不要）
- バック：軽量Webアプリフレームワーク + RDB（PostgreSQL／MySQL）
- AI連携：OpenAI/Anthropic API or 既存のkintoneにつなぐ場合はその限りで
- ホスティング：既存のオンプレ／VPS／クラウド、いずれでも可

> 既存のkintone資産がある前提で、**kintoneのカスタムビュー＋APIで実装する案**も別途比較検討します（ライセンス条件次第）。
